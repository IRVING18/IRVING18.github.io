一、出现原因、解决什么问题?

1、出现原因：高效保证线程安全的数据

场景在现(纯属个人理解，如有偏颇，望您不吝赐教)：

通常在解决线程安全问题中，使用的是Volatile/Atomic/Sychronized/Lock方式，但这些方法保证线程安全都是有性能牺牲的，降低系统吞吐量。
ThreadLocal的方案是不需要任何锁的干预的，它的思路其实很好理解，
tips：想象一个场景，如果我们想让每一个Thread线程都拥有自己的一套data数据，保证线程安全的同时，又可以让我们很方便的随时获取。

（1）线程安全的方案：

- 1、可以定义静态map数组方便随时获取，Thread当成key，data当成value存在里边。
- 2、然后在get()/set()时加上锁，来保证线程安全

（2）换一种思路，既然Thread之间操作相同数据有线程安全，我们何不直接将数据存到每一个Thread中呢？每个线程自己操作自己的data不就好了？然后再开放一个方法来set()/get()方便获取。那么最好的方法其实就是把这个data直接定义到Thread的成员变量上，然后再开放一个方法来set()/get()方便获取。事实上ThreadLocal就是这么做的，

  • 你可能要说，那我们直接extends Thread，定义成员变量data不就好了？但是如果我们每次都要去继承Thread，然后定义成员变量，那么代码的可维护性是不是就很低？而且要破坏Thread本身封装的开闭原则。
  • 所以正因为这种场景，jdk的ThreadLocal方案应运而生，它的本质就是在Thread中以成员变量的形式存储data。只是它做的比我们想的更完美，他是用Entry[k,v]数组的形式，扩展性更好。

2、ThreadLocal的作用

ThreadLocal是解决线程安全问题一个很好的思路，它通过为每个线程提供一个独立的变量副本解决了变量并发访问的冲突问题。
在很多情况下，ThreadLocal比直接使用synchronied同步机制解决线程安全问题更简单，更方便，且结果程序拥有更高的并发性。

二、用途场景，使用示例

在Java的多线程编程中，为保证多个线程对共享变量的安全访问，通常会使用synchronized来保证同一时刻只有一个线程对共享变量进行操作。
这种情况下可以将类变量放到ThreadLocal类型的对象中，使变量在每个线程中都有独立拷贝，不会出现一个线程读取变量时而被另一个线程修改的现象。
最常见的ThreadLocal使用场景为用来解决数据库连接、Session管理、Handler/Looper机制中。

三、原理、源码分析

THreadLocal原理：

1、在Thread中预置一个ThreadLocal.ThreadLocalMap成员变量，其中Entry[k,v]键值对数组，用于存储数据。
2、ThreadLocalMap是自定义用来专属的集合，功能类似map。
    Key 是ThreadLocal对象，
    value 是Object
3、通过ThreadLocal类，暴露对外的set()/get()方法

通过源码分析验证

1、Thread源码：持有存储键值对的成员变量threadLocals

2、ThreadLocalMap源码：存储键值对的类

K => ThreadLocal对象

V => Object

ThreadLocalMap是ThreadLocal的静态内部类

3、ThreadLocal源码：对外暴露get()/set()方法等

原理就是，

1、通过t = Thread.currentThread();获取当前操作的线程
2、map = t.threadLocals 直接获取当前Thread的成员变量
然后操作就都是Thread线程自己的东西了。

四、Hash冲突

与 HashMap 不同，ThreadLocalMap 结构非常简单，没有 next 引用，也就是说 ThreadLocalMap 中解决 Hash 冲突的方式并非链表的方式，而是采用线性探测的方式。

线性探测，就是根据初始 key 的 hashcode 值确定元素在 table 数组中的位置，如果发现这个位置上已经被其他的 key 值占用，则利用固定的算法寻找一定步长的下个位置，依次判断，直至找到能够存放的位置。

ThreadLocal 方式：

（1）ThreadLocal作为map的key，有默认的hash初始值怎么得来的？

  • 1、ThreadLocal对象，本身持有一个普通成员变量threadLocalHashCode = nextHashCode()；这个值即为每个ThreadLocal对象提供默认的hash值。在set()给map，时通过hashcode & (len - 1)设置到map中
  • 2、threadLocalHashCode是如果做到hash的呢？ThreadLocal本身持有，static的线程安全的静态变量nextHashCode = new AtomicInteger()。
  • 3、nextHashCode是通过CAS方式增长，nextHashCode.getAndAdd(HASH_INCREMENT);固定增长HASH_INCREMENT = 0x61c88647; 通过这种方式来提供hash算法

（2）key.threadLocalHashCode & (len-1) 发生hash冲突怎么处理？

  • 1、通过key.threadLocalHashCode & (len-1)计算得出的角标i，如果在map中已经存在，即为发生hash冲突
  • 2、THreadLocal处理很简单，直接将i+1，直到找到合适位置插入。
  • 3、当然这过程中，如果发现key，已经被GC回收了，还会去replaceStaleEntry()，释放一下map。

（3）ThreadLocalMap的扩容规则

  • 1、容量小于 2/3 就扩容
  • 2、每次扩容两倍

五、内存泄露原因？解决方案？

ThreadLocal 在 ThreadLocalMap 中是以一个弱引用身份被 Entry 中的 Key 引用的，因此如果 ThreadLocal 没有外部强引用来引用它，那么 ThreadLocal 会在下次 JVM 垃圾收集时被回收。这个时候 Entry 中的 key 已经被回收，但是 value 又是一强引用不会被垃圾收集器回收，这样 ThreadLocal 的线程如果一直持续运行，value 就一直得不到回收，这样就会发生内存泄露。

解决办法

  1. 使用完后记得remove

2.ThreadLocal自己提供了这个问题的解决方案。

每次操作set、get、remove操作时，会相应调用 ThreadLocalMap 的三个方法，ThreadLocalMap的三个方法在每次被调用时 都会直接或间接调用一个 expungeStaleEntry() 方法，这个方法会将key为null的 Entry 删除，从而避免内存泄漏。


3. 使用static修饰ThreadLocal

可以使用 static 修饰ThreadLocal，保证ThreadLocal为强引用，也就能保证任何时候都能通过ThreadLocal的弱引用访问到Entry的value值，进而清除掉 。

原因：根据可达性算法分析，类静态属性引用的对象可作为GC Roots根节点，即保证了ThreadLocal为不可回收对象

六、为什么有内存泄露风险，JDK仍然这么设计呢？

个人理解：

根据上边的分析，其产生内存泄漏的原因最直接的原因是ThreadLocal作为弱引用的key被回收了，导致[null,value]的键值对被ThreadLocalMap持有，而ThreadLocalMap被GCRoots 主线程持有，所以GC回收不了。

不难看出持有顺序是：GCRoots 主线程 -> ThreadLocalMap -> Entry[null , value] -> value

最终泄露的源头是，对Android来讲，主线程永不退出，那么主线程永远都是GC Roots，所以才会最终导致value的内存泄露。

所以，单纯的将key也设置成强引用，并不能解决问题，反而还多占用了key的内存值，导致k->v 都不能释放了。

故，这些场景只能是由开发者自己维护好 。
