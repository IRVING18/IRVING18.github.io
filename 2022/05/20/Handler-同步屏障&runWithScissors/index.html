一、同步屏障工作原理

（1）发送普通消息时

在发送消息时，

  • 1、当Handler().postMessage()/sendMessage()时最终都会调用enqueueMessage()，而在这里会将target设置为Handler本身；

    而且Handler.enqueuMessage()会调用MessageQueue.enqueueMessage()

  • 2、在MessageQueue中是不允许 target 属性为空的，否则会抛出异常；

（2）发送同步屏障时

在发送同步屏障时，有单独的方法：postSyncBarrier()

  • 1、postSyncBarrier() 的target没有设置，所以target 为 null
  • 2、设置当前Message到链表头部

（3）读取同步屏障时

我们知道Looper会无限循环获取MessageQueue中的Message数据，
所以来看Message.next()方法

  • 1、当读取到target == null就是读取到了同步屏障消息
  • 2、此时就会只执行异步消息了，即设置了mAsynchronous = true的消息；

二、同步屏障作用

主要作用就是提高设置mAsynchronous = true属性的消息优先级

例如：
在Choreography每16ms接收到onVysnc()屏幕信号时，就准备开始进行绘制了

  • 1、先发送同步屏障消息，暂停其他同步消息任务
  • 2、再发送异步消息到MessageQueue中，这个异步消息就会去执行onMeasure()、onLayout()、onDraw()的绘制任务

三、Handler的runWithScissors(Runnable r, long timeout)

（1）方法作用

在WMS初始化时用到了，就是一个异步转同步的方式；

Handler中一个@hint的隐藏方法：
它的作用是确保在IO线程中，执行一个期望在UI线程中执行一个任务；比如更新UI等操作；

（2）调用后具体分为两种情况：

  • 当前线程 == UI线程：那么会直接执行入参的Runnable任务
  • 当前线程 != UI线程：
    • 1、在UI线程执行任务
    • 2、当前线程会被wait()挂起，等待Runnable被执行完再notify()唤醒当前线程

（3）具体说明

实际上，它作为Handler的一个方法，就说明并不只是针对IO线程和UI线程；

我们自定义的两个线程同样生效，所以可以抽象成

  • Current Thread 当前线程（类比IO）
  • Target Thread 目标线程（类比UI）

同样分为两种情景：

  • Current Thread == Target Thread：那么会直接执行入参的Runnable任务
  • Current Thread ！= Target Thread：
    • 1、在Target线程执行任务
    • 2、Current线程会被wait()挂起，等待Runnable被执行完再notify()唤醒Current线程
