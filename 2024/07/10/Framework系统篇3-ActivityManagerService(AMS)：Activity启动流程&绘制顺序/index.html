tips：

1、为什么和zygoet通信是Socket呢？

答：

  • zygoet需要保障单进程单线程运行，否则会有死锁的风险；
    所以它没有使用binder机制，因为binder属于多线程运行；
  • 且这个Socket是系统级别的，端口号普通应用并不能访问，安全性是ok的；

2、如果是多线程操作时，fork会有问题？

fork原理：

  • 父进程内存拷贝到子进程
  • 子进程单线程启动；也就是父进程有多个线程时，子进程不会继承过来

若父进程是多线程时：

  • 1.线程中doit()会将mutex变量加锁
  • 2.此时fork子进程，
    • 那么子进程会拷贝mutex变量到自己的线程中，而且还是加锁的；
  • 3.此时父进程的多线程操作释放mutex时
    • 父进程的mutex释放了
    • 但是子进程的mutex并不会被释放，这就造成子进程死锁了；

（3）Choregrapher 回调TraversalRunnable任务过程；

Choreographer 的引入，主要是配合系统Vsync垂直同步机制（Android“黄油计划”中引入的机制之一，
1、协调APP生成UI数据和SurfaceFlinger合成图像，避免Tearing画面撕裂的现象）
2、给上层 App 的渲染提供一个稳定的 Message 处理的时机，也就是 Vsync 到来的时候 ，系统通过对 Vsync 信号周期的调整，来控制每一帧绘制操作的时机。Choreographer 扮演 Android 渲染链路中承上启下的角色：

  • 承上：负责接收和处理 App 的各种更新消息和回调，等到 Vsync 到来的时候统一处理。比如集中处理 Input(主要是 Input 事件的处理) 、Animation(动画相关)、Traversal(包括 measure、layout、draw 等操作) ，判断卡顿掉帧情况，记录 CallBack 耗时等；
  • 启下：负责请求和接收 Vsync 信号。接收 Vsync 事件回调(通过 FrameDisplayEventReceiver.onVsync )，请求 Vsync(FrameDisplayEventReceiver.scheduleVsync) 。

1、收到postCallback(Choreographer.CALLBACK_TRAVERSAL, mTraversalRunnable)

  • 通过key：CALLBACK_TRAVERSAL，将Runnable先存储到mCallbackQueues队列里；方便后续调用；
    • 注意这个Runnable就是上文中的TraversalRunnable

2、判断是否在主线程中；在主线程执行doScheduleVsync();

  • 若不在主线中，先发送MSG_DO_SCHEDULE_VSYNC 同步消息；
  • 将主线切换到主线程执行doScheduleVsync();
    • doScheduleCallback
      • scheduleVsyncLocked()
        • mDisplayEventReceiver.scheduleVsync();

3、mDisplayEventReceiver.scheduleVsync(); 注册FrameDisplayEventReceiver到c层

注册FrameDisplayEventReceiver接受者到c层，用于接收下一次的VSYNC垂直信号；

  • 每16ms会回调一次Receiver的#onVsync()方法；
  • 可通过hook此刻，记录startTime；用于记录卡顿情况

4、onVsync() 通过Handler发回主线程 => 执行doFrame()

  • 在doFrame()中会依次执行任务
    • CALLBACK_INPUT类型：输入源
    • CALLBACK_ANIMATION：动画
    • CALLBACK_INSETS_ANIMATION：
    • CALLBACK_TRAVERSAL：绘制，此时会执行传入的TraversalRunnable任务；也就回调到doTraversal()了；
    • CALLBACK_COMMIT：结束绘制
      • 可通过hook此处，记录endTime；用于计算丢帧数：(endTime - startTime) / 16ms

（4）RenderThread渲染

截止到目前，在ViewRootImpl中完成了对界面的measure、layout和draw等绘制流程后，用户依然还是看不到屏幕上显示的应用界面内容。

因为整个Android系统的显示流程除了前面讲到的UI线程的绘制外，

  • 1、界面还需要经过RenderThread线程的渲染处理，
  • 2、渲染完成后，还需要通过Binder调用“上帧”交给surfaceflinger进程中进行合成后送显才能最终显示到屏幕上。

也就是RenderThread线程会将view创建成RenderNode节点，最后合成bitmap对象设置给mSurface上，而mSurface就是通过WMS从SurfaceFlinger申请的内存空间画布；

RenderThread合成后写入mSurface中，SurfaceFlinger在另一个进程也会监听VSYNC信号，然后通过BufferQueue轮询获取mSurface对象，如果mSurface需要刷新，就在SurfaceFlinger中将数据合成帧，最后写入/dev/device硬件显示的内存空间中，也就展示到硬件屏幕上了；

所以在App进程接受VSYNC信号，会合成view节点刷入mSurface中；SurfaceFlinger也会监听VSYNC信号，处理mSurface数据合成帧；但他们都是独立进行的，所以可能会app在这次的VSYNC信号刷入了mSurface中，但是要等到下次SurfaceFlinger收到VSYNC信号时才会更新；因为他们是通过BufferQueue加锁处理的；

1、构建绘制命令树 updateRootDisplayList

构建绘制命令树的过程是从View控件树的根节点DecorView触发，递归调用每个子View节点的updateDisplayListIfDirty函数，最终完成绘制树的创建，简述流程如下：

  • （1）利用View对象构造时创建的RenderNode获取一个SkiaRecordingCanvas“画布”；
  • （2）利用SkiaRecordingCanvas，在每个子View控件的onDraw绘制函数中调用drawLine、drawRect等绘制操作时，创建对应的DisplayListOp绘制命令，并缓存记录到其内部的SkiaDisplayList持有的DisplayListData中；

简单来说：

它的职责就是负责接受来自多个来源的数据缓冲区，对它们进行合成，然后发送到显示设备。如下图所示：

从上图可以看出，其实SurfaceFlinger在Android系统的整个图形显示系统中是起到一个承上启下的作用：

  • 对上：通过Surface与不同的应用进程建立联系，接收它们写入Surface中的绘制缓冲数据，对它们进行统一合成。
  • 对下：通过屏幕的后缓存区与屏幕建立联系，发送合成好的数据到屏幕显示设备。

图形的传递是通过Buffer作为载体，Surface是对Buffer的进一步封装，也就是说Surface内部具有多个Buffer供上层使用，如何管理这些Buffer呢？答案就是BufferQueue ，下面我们来看看BufferQueue的工作原理：

1、BufferQueue机制

BufferQueue是一个典型的生产者-消费者模型中的数据结构。在Android应用的渲染流程中，应用扮演的就是“生产者”的角色，而SurfaceFlinger扮演的则是“消费者”的角色，其配合工作的流程如下：

  • （1）应用进程中在开始界面的绘制渲染之前，需要通过Binder调用dequeueBuffer接口从SurfaceFlinger进程中管理的BufferQueue 中申请一张处于free状态的可用Buffer，如果此时没有可用Buffer则阻塞等待；
  • （2）应用进程中拿到这张可用的Buffer之后，选择使用CPU软件绘制渲染或GPU硬件加速绘制渲染，渲染完成后再通过Binder调用queueBuffer接口将缓存数据返回给应用进程对应的BufferQueue（如果是 GPU 渲染的话，这里还有个 GPU处理的过程，所以这个 Buffer 不会马上可用，需要等 GPU 渲染完成的Fence信号），并申请sf类型的Vsync以便唤醒“消费者”SurfaceFlinger进行消费；
  • （3）SurfaceFlinger 在收到 Vsync 信号之后，开始准备合成，使用 acquireBuffer获取应用对应的 BufferQueue 中的 Buffer 并进行合成操作；
  • （4）合成结束后，SurfaceFlinger 将通过调用 releaseBuffer将 Buffer 置为可用的free状态，返回到应用对应的 BufferQueue中。

2、sync同步机制

Vysnc垂直同步是Android在“黄油计划”中引入的一个重要机制，本质上是为了协调BufferQueue的应用生产者生成UI数据动作和SurfaceFlinger消费者的合成消费动作，避免出现画面撕裂的Tearing现象。Vysnc信号分为两种类型：

  • （1）app类型的Vsync：
    • app类型的Vysnc信号由上层应用中的Choreographer根据绘制需求进行注册和接收，用于控制应用UI绘制上帧的生产节奏。根据【（3）Choregrapher 回调TraversalRunnable任务过程】的分析：应用在UI线程中调用invalidate刷新界面绘制时，需要先透过Choreographer向系统申请注册app类型的Vsync信号，待Vsync信号到来后，才能往主线程的消息队列放入待绘制任务进行真正UI的绘制动作；
  • （2）sf类型的Vsync:
    • sf类型的Vsync是用于控制SurfaceFlinger的合成消费节奏。应用完成界面的绘制渲染后，通过Binder调用queueBuffer接口将缓存数据返还给应用对应的BufferQueue时，会申请sf类型的Vsync，待SurfaceFlinger 在其UI线程中收到 Vsync 信号之后，便开始进行界面的合成操作。

五、高频问题

（1）每16ms会调用一次onDraw()方法吗？不是

并不一定会执行onDraw()绘制流程；
需要看在这个16ms内，有没有调用requestLayout()方法；

（2）invilidate()方法会立刻刷新吗？不会

不会，它需要等到下次VSYNC垂直信号到来的时候，才会执行；

（3）丢帧是因为这个吗？

是的，requestLayout()会通过mChoreography.post(CALL_TRAVELS)；
这里会记录状态，将任务放到了CallQueue中，

  • choreography会再c层的mFiling种注册recevier，以接收vsync回调；
  • onVsync()执行后，会执行queue中的任务；
  • 在commit任务时会计算：onVsync()信号开始到绘制完成的总时长；然后/16ms 比较看丢了几帧数据；

（4）VSYNC信号是固定16ms？

Vsync信号和屏幕刷新率相关；
60HZ的设备 => 1s 内发送60次 VSYNC信号；也就是16ms；

（5）Activity启动后View何时开始绘制（onCreate中还是onResume之后？）

上边已经描述很清楚了：onResume()之后；

ActivityThread核心方法：

  • handleLaunchActivity()
    • onAttach()创建window；通过Wms获取windowManager
    • onCreate()创建Decorview，
  • handleResumeActivity()
    • onResume()生命周期
    • 创建ViewRootImpl；
    • 添加到mGlobal.add(decorview)；注册到WMS中管理
    • ViewRootImpl.requestLayout()开启Choreography进行绘制；

（6）子线程能不能更新ui？

1、在RootViewImpl创建之前去更新是OK的，不会闪退；

在imageView.setImageResource(id)时，会调用invalidate()方法；
执行时会循环判断parent是否创建了；最终会调用到ViewRootImpl是否创建；

  • 创建了：会调用root.requestLayout() 走checkTherad()判断线程
  • 没创建：不会调用root.requestLayout() 中；也就不会触发线程判断逻辑
  • 注意root.requestLayout()和view.requestLayout()方法名相同，但不是一个；

那么为什么我们设置的resourceid还是能生效呢？

  • 因为虽然没有走到root.requsetLayout()去渲染UI，但是id已经设置给ImageView了，当root创建时会直接用到我们设置的id渲染；
