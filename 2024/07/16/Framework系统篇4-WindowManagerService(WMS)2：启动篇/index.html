一、概述

（1）WMS、Surface、SurfaceFlinger关系

WMS核心关系：

  • Surface: 代表画布
  • WMS: 添加window的过程主要功能是添加Surface,管理所有的Surface布局,以及Z轴排序问题;
  • SurfaceFinger: 将Surface按次序混合并显示到物理屏幕上;

（2）WMS整体认识

  • 一、WMS端：
    • WMS继承于IWindowManager.Stub, 作为Binder服务端;
    • WMS的成员变量mSessions保存着所有的Session对象,Session继承于IWindowSession.Stub, 作为Binder服务端;
    • 成员变量mPolicy: 实例对象为PhoneWindowManager,用于实现各种窗口相关的策略;
    • 成员变量mChoreographer: 用于控制窗口动画,屏幕旋转等操作;
    • 成员变量mDisplayContents: 记录一组DisplayContent对象,这个跟多屏输出相关;
    • 成员变量mTokenMap: 保存所有的WindowToken对象; 以IBinder为key,可以是IAppWindowToken或者其他Binder的Bp端;
  • 二、APP端：
    • ActivityRecord.Token extends IApplicationToken.Stub
    • 成员变量mWindowMap: 保存所有的WindowState对象;以IBinder为key, 是IWindow的Bp端;
    • ViewRootImpl.W extends IWindow.Stub
    • 一般地,每一个窗口都对应一个WindowState对象, 该对象的成员变量mClient用于跟应用端交互,成员变量mToken用于跟AMS交互.

二、startOtherServices()启动过程【SystemServer主线程】

（1）WMS.main()【启动”android.display”线程】

（2）new WindowManagerService() 创建WMS，【”android.display”线程】处理和SurfaceFilinger渲染

创建WMS的过程在“android.display”线程内操作的；

（3）wms.initPolicy()，启动【”android.ui”线程】

（4）PhoneWindowManager.init() 【”android.ui”线程】 处理InputManagerService 焦点变化、输入事件的分发

（5）runWithScissors()方法是用来同步执行代码的

注意这里一共涉及三条线程：

  • 1、SystemServer主线程，调用WindowManagerService.main()方法，启动display线程；
  • 2、”android.display”线程，由DisplayThread.getHandler().runWithScissors(new Runnable(){})启动
  • 3、”android.ui”线程，由PhoneWindowManager.init()中，UiThread.getHandler().runWithScissors(new Runnable() {})启动

这里涉及的runWithScissors方法是用来同步处理的，

5.1 Handler.runWithScissors()

5.2 BlockingRunnable.postAndWait()

由此可见, BlockingRunnable.postAndWait()方法是阻塞操作,

  • 就是先将消息放入Handler所指向的线程, 此处是指”android.ui”线程,
  • 由于该方法本身运行在”android.display”线程. 也就意味着”android.display”线程会进入等待状态,
  • 直到handler线程执行完成后再唤醒”android.display”线程.

那么PWM.init()便是运行在”android.ui”线程,属于同步阻塞操作.

三、以“android.display”线程为例，说明WMS和App的交互

在上述WMS启动过程中，并没有找到while()循环，然后等待处理消息的代码对吧？
因为WMS已经拆成两部分了

  • “android.display” 线程，处理和App的Surface关联
  • “android.ui” 线程，处理手势分发

都是通过Hanlder来进行维护的，

  • AMS中会调用WMS.setAppStartingWindow() 方法，调用过至WMS中来

四、总结

（1）android.ui、android.display 两条线程的作用？

  • android.display 线程
    • 配合 SurfaceFlinger 和显示缓冲区之间的交互，确保窗口内容能够正确渲染到显示设备上。
  • android.ui 线程
    • 配合 InputManagerService，分发用户的输入事件（例如触摸事件、键盘事件）到相应的窗口，并与输入系统协同工作（如 InputDispatcher、InputReader）。

（2）手势交互 -> Surface更新过程

  • （一）输入事件的分发过程

    • 1、InputDispatcher 和 InputReader：
      • InputReader 线程负责从输入设备（如触摸屏、键盘）读取原始输入事件，并将其转换为标准化的输入事件（如 MotionEvent、KeyEvent）。
      • InputDispatcher 线程接收 InputReader 处理后的标准化事件，并根据当前窗口焦点状态和窗口属性（如是否可接收触摸事件）将事件分发到正确的窗口。
    • 2、分发到应用的 Activity：
      • InputDispatcher 会将输入事件发送到应用程序中的 Window。应用端的 Activity 或 View 系统会接收到这些事件，并在 onTouchEvent() 或 onKeyEvent() 等方法中进行处理。

  • （二）应用端的事件处理与UI更新

    • 1、应用接收到输入事件后，通常会在 View 的事件回调方法中（如 onTouchEvent()）处理事件。如果事件导致了 UI 的变化（如按钮被点击，列表项被滑动等），应用会请求重新绘制相应的 View 或窗口。
    • 2、View 系统会将重绘请求提交给 Choreographer，Choreographer 负责调度下一帧的绘制，并将绘制请求添加到 VSYNC 信号触发的帧刷新周期中。

  • （三）窗口更新与 WMS 的交互

    • 1、当应用请求窗口内容的重绘时，ViewRootImpl 会将窗口内容绘制到 Surface 中（通常是一个 SurfaceTexture 或 GraphicBuffer），并将其提交给 SurfaceFlinger。
    • 2、此时，WindowManagerService（WMS）会感知到该窗口的内容或属性发生了变化。例如，窗口的大小、位置、层级等需要更新。

  • （四）android.display 线程的渲染更新

    • android.display 线程 负责与 SurfaceFlinger 的交互：
      • 它会将 Window 的变化通知 SurfaceFlinger，例如窗口的 Z 轴顺序、大小、位置等。
      • SurfaceFlinger 收到这些变化后，会进行合成（compositing），将多个窗口的内容绘制到帧缓冲区中，并最终将其渲染到屏幕上。
