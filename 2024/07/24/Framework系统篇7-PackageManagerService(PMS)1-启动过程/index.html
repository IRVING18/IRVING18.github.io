完整流程图

一、前述

1.1 先了解下PMS的作用

PackageManagerService(简称PKMS)，是Android系统中核心服务之一，管理着所有跟package相关的工作。
常见的比如：

  • 安装APK
  • 卸载APK
  • 获取Activity等四大组件的静态类名等

PKMS服务也是通过binder进行通信，IPackageManager.aidl由工具转换后自动生成binder的：

  • 服务端IPackageManager.Stub和
  • 客户端IPackageManager.Stub.Proxy，

  • Binder服务端：
    • 进程：运行在system_server进程中，提供服务
    • 实现：PackageManagerService继承于IPackageManager.Stub；
  • Binder客户端：
    • 进程：运行在app端，通过PackageManager类暴露实现方法，与服务端通信
    • 实现：
      • PackageManager的实现类是：ApplicationPackageManager(简称APM)
      • APM中持有成员变量mPM，继承于IPackageManager.Stub.Proxy; 即Binder的客户端proxy

1.2 从SystemServer进程调度PMS启动

在[Fw3-系统启动流程]一文中，我们已经知晓systemServer进程由Zygoet fork出来。

核心流程如下：

  • SystemServer.run()方法调度PMS执行
    • startBootstrapServices()中启动PMS.main()
    • startOtherServices()中继续调用PMS核心方法
      • mPMS.performBootDexOpt()
      • mPMS.systemReady()

1.2.2 在startBootstrapServices()中启动PMS.main()

1.2.3 在startOtherServices()中继续调用PMS核心方法：performBootDexOpt()、systemReady()

二、PMS.main()启动

PMS初始化时，有五个阶段，下边分别将这五个阶段

2.1 开始阶段（PMS_START）创建Settings保存app安装信息等

这个阶段源码很多，我们只记几个关键点：

  • 1、Settings：用于保存所有app的安装信息等
  • 2、创建Dex优化工具类：用于给dex包优化的工具
  • 3、SystemConfig：存储应用的权限等信息，如系统应用等
  • 4、在Data分区创建一些目录：如/data/data 等

2.1.1 Settings：用于保存所有app的安装信息等

Settings中存储的信息，
比如启动Activity时，获取Activity的绝对路径就在这里获取；

文件 信息
packages.xml 记录所有安装app的信息
packages-backup.xml 备份文件
packages-stopped.xml 记录系统被强制停止的文件
packages-stopped-backup.xml 备份文件
packages.list 记录应用的数据信息

2.1.2 SystemConfig：存储应用的权限等信息，如系统应用等

目录 作用
/system/etc/sysconfig 存放系统配置文件，定义系统服务和核心应用的行为及配置关键系统设置。 包括系统关键应用、默认应用设置、数据保留策略、节电白名单等。
/system/etc/permissions 存放权限定义文件，定义 Android 系统中权限的分配。 包括平台权限、权限分组、特定权限的分配等。
/oem/etc/sysconfig 供设备制造商（OEM）使用的系统配置目录，用于自定义系统配置。 可用于覆盖或扩展系统默认配置，如预装应用和系统服务行为。
/oem/etc/permissions 供 OEM 使用的权限配置目录，用于定义或扩展设备的权限配置。 可为设备特定应用授予额外权限，或定制权限策略。

2.1.3 在Data分区创建一些目录：如/data/data 等

创建的这些目录分类

变量 所对应目录 含义
mAppDataDir /data/data 存储应用的私有数据和配置文件，每个apk对应一个包名的目录
mAppInstallDir /data/app 用于存放已安装的app应用
mAppLib32InstallDir /data/app-lib 用于存放应用的本地库文件（即 .so 文件）
mAsecInternalPath /data/app-asec 存放在 Android 早期版本中使用的 ASEC 格式的应用
mUserAppDataDir /data/user 用于管理多用户环境中的用户数据，Android支持多用户
mDrmAppPrivateInstallDir /data/app-private 用于存放已安装的私有应用

2.2 扫描系统阶段（PMS_SYSTEM_SCAN_START）读取系统app文件

系统扫描阶段的主要工作有以下3点：

  • 创建/system的子目录，比如/system/framework、/system/priv-app和/system/app等等
  • 扫描系统文件，比如/vendor/overlay、/system/framework、/system/app等等目录下的文件。
  • 对扫描到的系统文件做后续处理。

扫描的目录：

目录 含义
app 存放系统App，包括了谷歌内置的App也有厂商或者运营商提供的App
framework 存放应用框架层的jar包
priv-app 存放特权App
lib 存放so文件
存放so文件 存放系统字体文件
media 存放系统的各种声音，比如铃声、提示音，以及系统启动播放的动画

2.3 扫描Data分区阶段（PMS_DATA_SCAN_START）读取用户App数据

主要作用：

  1. 扫描/data/app和/data/app-private目录下的文件。
  2. 遍历possiblyDeletedUpdatedSystemApps列表，
    • 注释1处如果这个系统App的包信息不在PMS的变量mPackages中，说明是残留的App信息，后续会删除它的数据。
    • 注释2处如果这个系统App的包信息在mPackages中，说明是存在于Data分区，不属于系统App，那么移除其系统权限。
  3. 遍历mExpectingBetter列表，
    • 注释3处根据系统App所在的目录设置扫描的解析参数，
    • 注释4处的方法内部会将packageName对应的包设置数据（PackageSetting）添加到mSettings的mPackages中。
    • 注释5处扫描系统App的升级包，最后清除mExpectingBetter列表。

扫描的目录：

目录 含义
app 存储用户自己安装的App
data 存储所有已安装的App数据的目录，每个App都有自己单独的子目录
app-private App的私有存储空间
app-lib 存储所有App的Jni库
system 存放系统配置文件
anr 用于存储ANR发生时系统生成的traces.txt文件

2.4 扫描结束阶段：PMS_SCAN_END

扫描结束结束阶段主要做了以下几件事（主要就是扫尾操作，清除缓存、判断APK版本等等）：

  • 如果当前平台SDK版本和上次启动时的SDK版本不同，重新更新APK的授权。
  • 如果是第一次启动或者是Android M升级后的第一次启动，需要初始化所有用户定义的默认首选App。
  • OTA升级后的第一次启动，会清除代码缓存目录。
  • 把Settings的内容保存到packages.xml中，这样此后PMS再次创建时会读到此前保存的Settings的内容。

主要作用就是启动PackageInstallerService服务

三、具体scanDirLI()扫描方法的作用

上述第二部分，将初始化分成了五个部分，其中

  • 扫描系统阶段（PMS_SYSTEM_SCAN_START）：对应系统app
  • 扫描Data分区阶段（PMS_DATA_SCAN_START）：对应普通app

两个阶段，会调用scanDirLI()方法，扫描各自文件目录下的app，具体看看这个方法

3.1 扫描的数据存在mActivities、mServices、mSettings中

成员变量 存储内容 核心功能
mActivities
  • 存储每个 Activity 的路径（类名）
  • 与 Activity 相关的 IntentFilter 信息
 通过 Intent 解析找到匹配的 Activity，并进行启动
mSettings
  • 系统中所有已安装应用的配置信息
  • 每个应用的组件（Activity、Service、Receiver、Provider）的配置信息
  • 应用的权限声明及授权信息
  • 应用的签名信息、安装路径、版本号等
 管理并持久化存储应用程序的配置信息

四、PMS.performBootDexOpt()优化（系统app、普通app）的dex文件

主要目的：

  • 1、筛选出需要dex优化的app
  • 2、遍历调用dexopt操作优化

4.1 PKMS.filterRecentlyUsedApps()过滤出需要dexopt优化的app（最近一周使用过的app）

获取最近使用的app,其中mDexOptLRUThresholdInMills：

  • 对于Engineering版本（即测试版），则只会对30分钟之内使用过的app执行dex优化；
  • 对于用户版本，则会将用户最近一周内使用过的app执行dex优化；

4.3 由mInstaller服务承接执行dexopt优化

mInstaller也是个系统服务，和PMS同级别，只是它的名字没叫Service；

具体optdex优化的源码就不看了，主要会做如下操作：

优化步骤 详细描述
转换为机器代码 将 DEX 文件转换为 ODex（Dalvik）或 OAT（ART）文件，生成适合设备的二进制机器码。
字节码分析和优化 进行方法内联、常量折叠、循环展开等字节码优化，以提高执行效率。
类加载和预验证 预先加载并验证类结构，减少运行时的类加载和验证开销。
预编译（AOT 编译） 在 ART 环境中使用 AOT 编译技术，将 DEX 字节码提前编译为机器码，减少运行时的编译负担。
JIT 编译优化 在运行时对频繁执行的代码路径进行 JIT 编译，进一步提升性能。
多种架构优化 根据设备的处理器架构（ARM、x86、MIPS 等）生成优化后的机器码，充分利用硬件特性。
减少内存占用 优化后的代码减少内存碎片并提高代码加载速度，降低运行时的内存使用。
共享库和交叉引用优化 优化应用程序中的共享库和交叉引用部分，减少代码重复，提高系统效率。
代码缓存 生成的 OAT 文件缓存到 /data/dalvik-cache，启动时直接加载优化后的代码。
运行时触发条件 触发条件包括应用安装/更新、系统更新、以及特定电源状态下（如充电时）的优化操作。

五、总结

  • （1）从SystemServer启动上说
    • 1、startBootstrapServices()中启动PMS.main()
    • 2、startOtherServices()中继续调用PMS核心方法：
      • performBootDexOpt()、systemReady()
  • （2）PMS.main()方法主要分为五个阶段
    • 1、开始阶段：创建Settings保存app安装信息等，开启文件目录
      • 1.1 Settings：用于保存所有app的安装信息等
      • 1.2 SystemConfig：存储应用的权限等信息，如系统应用等
      • 1.3 在Data分区创建一些目录：如/data/data 等
    • 2、扫描系统阶段：读取系统级别app、Framework文件
    • 3、扫描Data分区阶段：读取用户级别App文件
    • 4、扫描结束阶段：扫尾操作，清除缓存啥的
    • 5、准备阶段：启动PackageInstallerService服务，用于apk安装的服务
  • （3）其中第2、3阶段，会读取系统、普通app的Activity等四大组件信息，以及mSettings会存储app的签名、权限等信息
    • scanDirLI()方法
  • （3）performBootDexOpt()过滤最近一周的app进行dexopt优化
