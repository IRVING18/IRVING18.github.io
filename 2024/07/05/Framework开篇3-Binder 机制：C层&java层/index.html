一、Binder c层、java层结构

1、c层结构：BnInterface(服务端)/BpInterface(客户端)；BBinder(服务端)/BpBinder(客户端)

参考链接：这俩是同一篇文章，不知道哪个是原创
https://paul.pub/android-binder-cpp/
https://www.cnblogs.com/hellokitty2/p/15088265.html

C层代码在Framework层的native目录下：

Binder Framework的C++部分，头文件位于这个路径：
/frameworks/native/include/binder/
实现位于这个路径：
/frameworks/native/libs/binder/

可通过线上看源码：
http://aospxref.com/android-8.0.0_r36/xref/frameworks/native/libs/binder/include/binder/Binder.h

![image.png](https://raw.githubusercontent.com/IRVING18/pic/master/Framework开篇3-Binder 机制：C层&java层_1.jpg)

主要结构：
libBinder源码中，将实现分为Proxy和Native两端。

  • Proxy对应了上文提到的Client端，是服务对外提供的接口。
  • Native是服务实现的一端，对应了上文提到的Server端。
  • 类名中带有小写字母p的（例如BpInterface），就是指Proxy端。类名带有小写字母n的（例如BnInterface），就是指Native端。

Proxy代表了调用方，通常与服务的实现不在同一个进程，因此下文中，我们也称Proxy端为“远程”端。
Native端是服务实现的自身，因此下文中，我们也称Native端为”本地“端。

类名 说明
BpRefBase RefBase的子类，提供remote()方法获取远程Binder
IInterface Binder服务接口的基类，Binder服务通常需要同时提供本地接口和远程接口
BpInterface 远程接口的基类，远程接口是供客户端调用的接口集
BnInterface 本地接口的基类，本地接口是需要服务中真正实现的接口集
IBinder Binder对象的基类，BBinder和BpBinder都是这个类的子类
BpBinder 远程Binder，这个类提供transact方法来发送请求，BpXXX实现中会用到
BBinder 本地Binder，服务实现方的基类，提供了onTransact接口来接收请求
ProcessState 代表了使用Binder的进程
IPCThreadState 代表了使用Binder的线程，这个类中封装了与Binder驱动通信的逻辑
Parcel 在Binder上传递的数据的包装器

C 层简单关系如下，

上层接口

  • RefBase 基类，提供最上层抽象方法
  • BpRefBase RefBase的子类，
    • 提供 remote() 方法获取远程Binder的句柄
  • IInterface
    • 1、作用：Binder服务的接口基类，为远程接口BpInterface和本地接口BnInterface提供公共抽象方法；
    • 比如BnInterfacce和BpInterface都要一个foo()方法，那么通过IInterface上层接口来定义就更好得管；但是通常它还有一个通过宏实现的IXXX接口，比如IActivityManager；通过这个子接口管理公共方法foo()
    • 2、 asBinder() 抽象方法；这个方法用于获取Binder对象，
      • 在BnInterface中实现就返回本地binder、
      • 在BpInterface中实现就返回远程binder；
      • 这样在代码中便可以直接通过IXXX::asBinder方法获取到对应IBinder对象，不用区分本地还是远程的，方便很多；
    • 3、通过宏会去实现 asInterface() ，用于获取Binder对象，
      • 当前进程直接通过descriptor获取，
      • 其他进程通过ServiceManager.getService(“power服务name”)获取handle句柄，然后new BpInterface(handle) 创建；构造函数会根据handle解析出服务端Binder对象；
      • 简单来说，handle是服务端Binder对象在Binder驱动中的标识，客户端通过这个handle，可以在Binder驱动中找到对应的服务端Binder对象，从而实现跨进程通信。

远程接口，即提供给客户端的远程代理：

  • BpBinder 远程Binder，
    • handle 是BBinder在启动binder驱动时，注册进去的唯一值；BpBinder可以根据handle直接在binder驱动中传输数据给BBinder；
    • uint32_t code值，用来区分接口方法的，根据code值对应不同的调用方法
    • transact() 发送数据
  • BpInterface 远程接口的基类，远程接口是供客户端调用的接口集
    • 继承IXXX，也就是继承IInterface，用来实现自定义接口方法；通常是和BpBinder、BnBinder中code相对应的方法；
    • 实现onAsBinder()方法，注册返回远程Binder对象；

本地接口，即服务端：

  • BBinder 本地Binder，服务实现方的基类，
    • onTransact() 接收数据
    • uint32_t code值，用于区分接口方法，和上边BpBinder中对齐；例如：add接口code设为1，minus接口code设为2，multiple接口code设为3，divide接口code设为4，等等
  • BnInterface 本地接口的基类，本地接口是需要服务中真正实现的接口集
    • 继承IXXX，也就是继承IInterface，用来实现自定义接口方法；通常是和BpBinder、BnBinder中code相对应的方法；
    • 实现onAsBinder()方法，注册返回本地Binder对象；
  • 会被注册到ServiceManager中，对应一个唯一svcinfo结构：包含name和handle

经过上边的分析，现在可以更好的理解客户端、服务端了，

客户端（远程服务）：

  • 是一个提供给客户端调用的代理proxy，即BpInterface的子类BpXXX（比如BpAcitityManger）；
  • code码和接口方法一一对应，同时对应服务端的接口方法；
  • 在通信时，
    • IActivityManager.asInterface()，先从ServiceManager通过name获取服务端的实体sp<IBinder>（svfinfo包含handle信息）
      • 在ActivityManger进程直接通过descriptor获取到实体了
      • 在其他进程，new BpActivityManager(sp<IBinder>)创建对象
    • 服务中remote()创建BpBinder(handle)对象，并持有引用
    • 调用remote()->transact()方法来发送数据，BpBinder把handle句柄、code、数据传给binder驱动，就可以直接和BBinder服务端通信了；

服务端（本地服务）

  • 是一个Binder服务的数据处理器，即BBinder->BnInterface->BnXXX(BnActivityManger)->XXXService(ActivityMangerService)；
  • 服务启动
    • open_driver()
      • open /dev/binder驱动，并注册唯一handle
      • mmap() 开辟内核空间
  • 启动时会通过ProcessState::self()
    • 注册到ServiceManager中，提供唯一name和handle
    • 创建线程池
    • IPCThreadState创建主线程，join到线程池中
  • code码和接口方法一一对应，客户端调用时会通过code码匹配相应接口方法；
  • 在onTransact()方法中，处理数据
  • 在通讯时，BpBinder调用transact(handle,code..)，通过给binder驱动handle，直接就能传到BBinder服务里，并走到onTransact()方法里；

这个过程如下图：
![image.png](https://raw.githubusercontent.com/IRVING18/pic/master/Framework开篇3-Binder 机制：C层&java层_2.jpg)

Binder框架，除了上边的规则之外，还有一些其他规则：
以ActivityManger为例：

  • getInterfaceDescriptor ：用于区分各个本地服务的标识，Binder服务会在类中定义static const android::String16 descriptor，然后在getInterfaceDescriptor方法中返回这个常量。
  • asInterface() ：为了便于调用者获取到接口对象，IXXX中会实现一个默认方法，android::sp<IXXX> asInterface返回该基类的对象；简单说就是用来获取IXXX（IActivityManger）对象的，只是在不同进程获取方式不同；
    • 创建BnActivityManager时，先通过ServiceManager去获取BBinder的sp<IBinder>实体：其实包含svginfo（name、handle）
    • 1、若是在当前ActivityManger服务端进程，直接就根据descriptor返回当前binder了
    • 2、若是在其他进程，通过handle创建BpBinder(handle)，BpBinder有了handle，就能通过binder驱动去transact()发送数据了；
  • 这些方法在c层，是通过宏来简化实现的，开发者只要在接口基类（IXXX）头文件中，使用DECLARE_META_INTERFACE宏便完成了需要的组件的声明。然后在cpp文件中使用IMPLEMENT_META_INTERFACE便完成了这些组件的实现。

2、关于Binder传递数据的大小限制：BINDER_VM_SIZE = 1M - 8k = 1016KB

由于Binder的数据需要跨进程传递，并且还需要在内核上开辟空间，因此允许在Binder上传递的数据并不是无无限大的。mmap中指定的大小便是对数据传递的大小限制：

这里我们看到，在进行mmap的时候，指定了最大size为BINDER_VM_SIZE，即 1M - 8k的大小。 因此我们在开发过程中，一次Binder调用的数据总和不能超过这个大小。

对于这个区域的大小，我们也可以在设备上进行确认。这里我们还之前提到的system_server为例。上面我们讲解了通过procfs来获取映射的内存地址，除此之外，我们也可以通过showmap命令，来确定这块区域的大小，相关命令如下：

angler:/ # ps  | grep system_server
system    1889  526   2353404 135968 SyS_epoll_ 72972eeaf4 S system_server
angler:/ # showmap 1889 | grep "/dev/binder"
    1016        4        4        0        0        4        0        0    1 /dev/binder
这里可以看到，这块区域的大小正是 1M - 8K = 1016k。

Tips: 通过showmap命令可以看到进程的详细内存占用情况。在实际的开发过程中，当我们要对某个进程做内存占用分析的时候，这个命令是相当有用的。建议读者尝试通过showmap命令查看system_server或其他感兴趣进程的完整map，看看这些进程都依赖了哪些库或者模块，以及内存占用情况是怎样的。

4、服务的获取

在服务已经发布之后，客户端该如何获取其服务接口然后对其发出请求调用呢？

很显然，客户端应该通过BpPowerManager的对象来请求其服务。但看一眼BpPowerManager的构造函数，我们会发现，似乎没法直接创建一个这类的对象，因为这里需要一个sp<IBinder>类型的参数。

那么这个sp<IBinder>参数我们该从哪里获取呢？

BpBinder其实是包含了一个指向Server的句柄，所有的请求发送出去的时候都需要包含这个 句柄 作为一个标识。

而想要拿到这个句柄，我们自然应当想到ServiceManager。我们再看一下ServiceManager的接口自然就知道这个sp<IBinder>该如何获取了：

在addService时，注册的字符串“power”来获取sp<IBinder>，其中包含handle句柄信息；

然后通过defaultServiceManager()去获取ServiceManager对象，就可以创建BpPowerManager对象了；

到这，其实我们已经跑完了整个流程，

  • 1、从Binder服务端创建，
    • open /dev/binder驱动
    • mmap 开辟内核空间映射
    • 重写onTransact()方法，处理code对应的接口方法
  • 2、服务发布到ServiceManager
    • 1、addService(“power”) 注册给ServiceManager
    • 2、创建线程池，并启动PowerManager的主线程
  • 3、客户端获取服务
    • 1、通过“power”字符串在ServiceManager查找sp<IBinder>数据（包含handle）
    • 2、new BpPowerManager(sp<IBinder>) 获取到Binder的proxy代理，就能通过remote()->transact()来发送数据了

但这里有一个需要注意的地方，
就是在获取BpPowerManager的过程中，

  • 我们刚刚是通过ServiceManager查到handle，来远程创建proxy代理对象的；
  • 实际中，获取BpPowerManager时，可能就在PowerManager的进程中，也就是进程内自己调用时，显然并不需要去走通过handle来跨进程获取proxy对象；

这里在宏中asInterface()中给了解释：

  • 在当前进程中，直接通过descriptor来获取BpPowerManager对象；
  • 在其他进程中，通过ServiceManager获取的sp<IBinder>中的handle句柄，new BpPowerManager(sp<IBinder>)对象，remote()会创建BpBinder(handle)；然后通过remote()->transcat(handle,code,传输数据)就能通过binder驱动给BBinder进程发送数据了；

六、总结

1、简述binder启动和通信过程

binder运行流程：
（一）Binder服务端启动

  • 1、Binder服务端创建open_driver()
    • 通过open_binder() 开启binder驱动
    • 通过mmap() 开辟内存映射，用于接收数据;1016KB
    • 重写onTransact()方法
  • 2、发布服务，ProcessState单例创建
    • 将Binder服务注册到ServiceManager中，addService(“power”)
    • 启动线程池
    • 启动主线程，在onTransact()方法中接收数据；join到线程池中

（二）客户端

  • 1、通过“power”字符串在ServiceManager查找sp<IBinder>数据（包含handle）
  • 2、new BpPowerManager(sp<IBinder>) -> remote()持有new BpBinder(handle) 获取到Binder的proxy代理，就能通过remote()->transact()来发送数据了

注意：

  • 1、客户端拿到的的是服务端的BpXXX->remote()持有BpBinder(handle)，这些都是通过handle new出来的对象；
  • 2、new的时候传入的handle，在BpBinder中就可以直接传给binder驱动，驱动中会去根据handle翻译找到BBinder服务端进程来通信；

（三）ServiceManager的特殊性

ServiceManager也是个Binder服务端；
那客户端是怎么获取到它的remote远程服务端地址呢？

答案是在Android系统启动时，

  • init.rc中，启动ServiceManager、zygote
  • 并将ServiceManager的Binder服务端，注册到特殊位置上
  • 所以在普通客户端获取ServiceManager时，可以直接通过defaultServiceManager()方法获取

3、Binder通信时的基础参数

① 控制类协议：如注册成ServiceManager

控制协议是进程通过ioctl(“/dev/binder”) 与Binder设备进行通讯的协议，该协议包含以下几种命令：

命令 说明 参数类型
BINDER_WRITE_READ 读写操作，最常用的命令。IPC过程就是通过这个命令进行数据传递 binder_write_read
BINDER_SET_MAX_THREADS 设置进程支持的最大线程数量 size_t
BINDER_SET_CONTEXT_MGR 设置自身为ServiceManager 无
BINDER_THREAD_EXIT 通知驱动Binder线程退出 无
BINDER_VERSION 获取Binder驱动的版本号 binder_version

② 驱动类协议：BC/BR

  • BC请求码：BINDER_COMMAND_PROTOCOL：描述了进程发送给Binder驱动的命令
  • BR响应码：BINDER_RETURN_PROTOCOL ：描述了Binder驱动发送给进程的命令

BC类型如下：

命令 说明 参数类型
BC_TRANSACTION Binder事务，即：Client对于Server的请求 binder_transaction_data
BC_REPLY 事务的应答，即：Server对于Client的回复 binder_transaction_data
BC_ENTER_LOOPER 通知驱动主线程ready void

BR类型如下：

命令 说明 参数类型
BR_OK 操作完成 void
BR_TRANSACTION 通知进程收到一次Binder请求（Server端） binder_transaction_data
BR_REPLY 通知进程收到Binder请求的回复（Client） binder_transaction_data
BR_TRANSACTION_COMPLETE 驱动对于接受请求的确认回复 void
BR_SPAWN_LOOPER 通知Binder进程创建一个新的线程 void

  • 一次拷贝
    • 由BBinder服务端开启mmap内存映射

      • 和服务端不在同一进程：new一个出来，new BpActivityManager(handle) -> new BpBinder(handle)

chatGpt画的图，不太准确，但是帮助理解

整个通信过程如下：

Client进程分配共享内存（mmap），用于存储请求数据和接收返回结果。

Client进程将请求数据（如调用的方法、参数等）写入共享内存。

Client进程通知Binder驱动，请求已准备好。

Binder驱动将请求信息（包括共享内存地址）传递给Server进程的Binder对象。

Server进程的Binder对象通过AMS将请求分发给目标服务。

目标服务处理请求。

目标服务将处理结果写入共享内存。

Client进程从共享内存中获取结果。

Client进程通知Binder驱动，结果已获取。

Binder驱动将结果返回给Client进程。

通过这个过程，包括使用mmap分配的共享内存，Android中的两个进程可以高效地进行通信。共享内存减少了数据拷贝的开销，提高了通信效率。同时，AMS作为中心化的服务管理组件，负责调度和管理各种服务，确保系统资源的合理分配。
